## Biggest issues / easy improvements
### 1) Tests import from `src` directly (this is a smell)
Current test:
```py
from src import main as app_main
```
That bypasses the *installed package* and can hide packaging/import mistakes. Prefer testing the importable package:

- Move your “entry point” code into the package, e.g. `src/python_minimal_boilerplate/cli.py`
- Tests import:
```py
from python_minimal_boilerplate.cli import main
```

Right now you have both `src/main.py` (script-ish) and `src/python_minimal_boilerplate/main.py` (library-ish). I’d consolidate: keep “library” under the package, and make CLI call into it.

### 2) `requires-python = ">=3.14"` is probably too strict for a template
Unless you *really* want to force 3.14-only, I’d set this to something like `>=3.11` or `>=3.12`. Templates get copied; a strict floor reduces adoption a lot.

### 3) PyPI metadata is minimal
For publishing you’ll want at least:
- `authors` / `maintainers`
- `license` (or `license-files`)
- `classifiers`
- `urls` (`Homepage`, `Repository`, `Issues`, `Changelog`)
- better `description`

Not required for installation, but very worth it.

## Should you add PyPI + CLI boilerplate? Yes—if “uvx.sh” is your target
If you want people to run your tool as a one-liner (via `uvx` or `uv tool run`), then packaging it as a PyPI distribution with an entry point is exactly the right move.

### The key piece: `project.scripts`
In `pyproject.toml` add something like:

```toml
[project]
name = "python-minimal-boilerplate"
# …

[project.scripts]
yourcli = "python_minimal_boilerplate.cli:main"
```

Then users can run:
- `uvx your-package-name` (common pattern)
- or `uv tool run your-package-name` (depending on how they invoke tools)

(Your actual command name is the *left side* of `project.scripts`, not necessarily the package name.)

### Suggested file layout for that
- `src/python_minimal_boilerplate/cli.py` containing `def main() -> None: ...`
- `src/python_minimal_boilerplate/__main__.py` (optional but nice) so `python -m python_minimal_boilerplate` works too

## Publishing boilerplate I’d add to the template
1) **GitHub Actions release workflow** (build + publish)
   - Use PyPI “Trusted Publishing” (OIDC) so no API tokens in secrets.
2) **Just recipes**
   - `just build` → `uv build`
   - `just publish-test` → publish to TestPyPI
   - `just publish` → publish to PyPI (probably gated on tags)
3) **Versioning strategy**
   - Either manual bump in `pyproject.toml`
   - Or adopt `hatch-vcs` / `setuptools-scm` style (auto from git tags). (Up to you; manual is simplest for a template.)
4) **Changelog discipline**
   - Keep a `CHANGELOG.md` and update it per release (or add towncrier if you want automation).
